# v0.3.0 Release Cut Plan

> **Status**: planned. **Target**: after Phase 4b + 4c blocks land.
> **Type**: minor with breaking-change. **Backwards-compat window**: closed.

v0.3.0 is the first release in the consolidated-review sequence that
makes a **breaking** change. Operators upgrading from v0.2.x must
perform a one-time migration before the upgrade completes.

---

## Breaking changes

### S6 — Encryption-key auto-generation removed

- **v0.2.11 (shipped)**: stderr deprecation warning at startup when
  the auto-generated `{dataDir}/shellyadmin.key` path triggers.
- **v0.3.0 (this cut)**: the path is removed entirely.
  `cmd/shellyctl/main.go` `loadEncryptionKey()` panics at startup if
  neither `SHELLYADMIN_ENCRYPTION_KEY` nor
  `SHELLYADMIN_ENCRYPTION_KEY_FILE` is set.

The panic message includes the path the operator should set, a
pointer to ADR-0013, and a hint to the `shellyctl` migration recipe.

**Migration recipe** (operator-side, documented in ADR-0013):

```bash
# 1. Read the auto-generated key from the existing container:
docker exec shellyadmin cat /data/shellyadmin.key > ./shellyadmin.key

# 2. Move it to a path outside the data volume (Docker secret, NixOS
#    secret store, sops-encrypted file in the homelab config repo).

# 3. Update compose .env to point at the new path:
SHELLYADMIN_ENCRYPTION_KEY_FILE=/run/secrets/shellyadmin_encryption_key

# 4. Pull v0.3.0, recreate stack.
```

Operators who skip the migration get a clear startup error; the
container exits 1 with the recipe in stderr.

### `runtime_locks` enforcement (ADR-0015)

v0.3.0 introduces `internal/db/migrations/030_runtime_locks.sql` (the
in-process state lock table). At startup the service:

1. Reads `runtime_locks` WHERE `key='primary'`.
2. If a row exists with a `acquired_at` < 5 minutes old, exits 1
   with a clear error naming the other instance's hostname/pid.
3. Otherwise upserts a fresh row + starts a heartbeat goroutine.
4. On graceful shutdown, deletes the row.

Single-instance operators see no change. Operators who tried (or
will try) to run two containers against one SQLite see the explicit
refusal instead of subtle bugs.

A `shellyctl unlock --force` subcommand provides the manual recovery
path for the rare case where a kill-9'd previous container left a
stale lock and the operator does not want to wait 5 minutes.

## Release flow

### Pre-flight (1 hour)

1. Make sure CI is green on every Phase-4b/4c sub-PR.
2. Run `go run ./cmd/modelschema` and commit any drift. The schema
   gains TOTPState + PAT entries.
3. Verify v0.2.x deployment still boots after upgrading the
   encryption key out of `/data` (manual smoke on a test container
   before tagging).
4. Update CHANGELOG with the consolidated 0.3.0 entry pulling from
   the sub-PR changelogs.
5. Bump VERSION, web/package.json, web/package-lock.json.

### Tag + push (15 minutes)

```bash
git commit -m "Release v0.3.0 - ..." VERSION web/package.json web/package-lock.json CHANGELOG.md
git tag v0.3.0
git push origin main v0.3.0
```

### Post-flight (1-2 hours)

1. Watch the publish-image workflow.
2. Verify cosign signing succeeds + Trivy scan clears on the new image.
3. On docker.home.lan: pull the new image, verify with cosign, run
   the migration recipe (move the encryption key out of /data),
   recreate the stack.
4. Verify Login → Devices → /ready returns the v0.3.0 version label.

### Rollback path

The migration is one-way **at the encryption key level**: the old
auto-generated key file is still in `/data` (the operator copied it
out, did not delete it). To roll back to v0.2.x:

1. Pull the previous v0.2.x image tag.
2. Recreate the stack without the new `SHELLYADMIN_ENCRYPTION_KEY_FILE`
   env var. v0.2.x finds the original `/data/shellyadmin.key` and
   continues.

Operators who deleted `/data/shellyadmin.key` after the migration
cannot roll back without restoring it from their key-management
store. Document this prominently in the upgrade notes.

## Excluded from v0.3.0 (queued for v0.4 / v1.0)

- ~~**T2 WebAuthn / passkeys** (XXL; depends on T1)~~ — **dropped (2026-05-20).**
  WebAuthn requires a secure context (HTTPS); prod is trusted-LAN plain HTTP on
  `docker.home.lan`, so passkeys can't run without a TLS reverse-proxy rebuild.
  For a single-operator internal tool the phishing-resistance payoff doesn't
  justify that infra cost — password (DB-backed since v0.4.0) + TOTP 2FA + PATs
  already cover the threat model. Revisit only if the tool goes multi-user or
  internet-exposed (which needs HTTPS anyway).
- ~~**T4 HSM/PKCS11 key provider** (XXL; depends on S6)~~ — **dropped
  (2026-05-20).** Would move `SHELLYADMIN_ENCRYPTION_KEY` out of env/file into a
  TPM/YubiHSM/KMS via PKCS#11 (secretbox becomes an HSM-backed interface). XXL
  effort for a threat S6/ADR-0013 already largely closed (key lives outside the
  DB backup boundary). Near-zero value without actual HSM hardware in the
  homelab. Revisit only if such hardware exists and the operator wants it.
- **T7 `/api/v1/*` prefix mounting** (the actual route shim; the
  policy doc is in `docs/ARCHITECTURE.md` already) — **explicitly v1.0-gated
  (decided 2026-05-20).** Pure plumbing with no payoff until a breaking
  `/api/v2` exists; for a single-operator tool breaking changes are absorbed by
  updating the one SPA + own scripts in lockstep. Do it AT the v1.0 cut (alias
  `/api/*` → `/api/v1/*` + `Deprecation` header), not before. ~½–1 session
  incl. tests; touches 54 documented routes + ~51 client call sites
  (`api.ts` 45, `shellyctl` 6).
- **T8 Playwright E2E test** (L; ships with v0.3.x once M2 stabilises)
- ~~**T9 external pen test** (extern; should run before v1.0 cut)~~ —
  **skip (2026-05-20)** unless the tool is ever internet-exposed. A paid
  third-party web-app pentest (typically four figures) buys little marginal
  assurance over the existing internal threat work (STRIDE ADR-0012, the
  consolidated review) for a single-operator LAN tool. Cheap substitute if
  curious: a self-run OWASP ZAP / nuclei scan against the instance.
- ~~**T12 standalone-binary signing via goreleaser** (L; docs in
  DEPLOYMENT.md already)~~ — **skip (2026-05-20)** until standalone binaries
  are actually distributed. The Docker image is ALREADY cosign-signed +
  Trivy-scanned in CI; T12 only covers the non-Docker binary channel, which
  this project doesn't publish. No consumer until binaries land on the GitHub
  Releases page.

These are tracked in the `Unreleased` section of CHANGELOG.md and
in the issue tracker as the v0.4+ roadmap.

## v0.3.0-rc cadence

Before v0.3.0 final, plan **at least one rc**:

- `v0.3.0-rc1` after all Phase-4b/4c PRs are merged.
- Run the rc on docker.home.lan for >= 1 week.
- Cut v0.3.0 once the rc has had a real-world soak.

The Phase-4c TOTP change is the most user-visible; the rc soak is
mostly there to catch operator-side workflow surprises (lost device,
backup-code recovery, etc.) before the breaking-change release.
